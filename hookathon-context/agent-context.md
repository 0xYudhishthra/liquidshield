## 1. Background — Why We Pivoted

The original Aqua0 model used **1inch Aqua** (SwapVM-based strategies, per-user vault accounts via `Account.sol` / `AccountFactory.sol`). That code still lives in `backend/`, `contracts/`, `web-app/` and is **untouched / preserved**.

For the **Uniswap V4 Hook Incubator / Hookathon**, we pivoted to a new model:

- **Aqua0 is now a shared liquidity layer** that feeds multiple Uniswap V4 pools simultaneously.
- Each V4 pool has the **Aqua0Hook** as its hook contract.
- On every swap, `beforeSwap` virtually injects shared liquidity, the swap executes via V4 flash accounting, then `afterSwap` removes the virtual liquidity — only the net swap delta actually moves tokens.
- Users deposit funds into a **single shared contract** (`SharedLiquidityPool`) and choose which pool + tick ranges to allocate to (like a Uniswap V3/V4 LP UI).

All hookathon code is **isolated in three new directories:**

| Directory | Purpose |
|---|---|
| `contracts-hookathon/` | Solidity contracts (SharedLiquidityPool, Aqua0Hook, deploy scripts) |
| `backend-hookathon/` | Hono API that serves calldata + reads on-chain state |
| `app-hookathon/` | Next.js frontend (pool marketplace, LP dashboard, deposit/position UI) |

## 2. Architecture

### Smart Contracts

```
User
 │  deposit(token, amount)
 ▼
SharedLiquidityPool          ← holds ALL user tokens in one contract
 │  addPosition(poolKey, tickLower, tickUpper, liquidity, t0Amount, t1Amount)
 │  removePosition(...)
 │  getAggregatedPositions(poolId) → RangeInfo[]
 │  settleSwapDelta(token, delta)   ← called by hook only
 ▼
Aqua0Hook  (IHooks)
 │  beforeSwap → for each active range: poolManager.modifyLiquidity(+delta)
 │  afterSwap  → for each active range: poolManager.modifyLiquidity(-delta)
 │               net delta settled via SharedLiquidityPool
 ▼
Uniswap V4 PoolManager
```

### Key Design Decisions

| Decision | Choice | Why |
|---|---|---|
| Token custody | `SharedLiquidityPool` (never PoolManager) | Single contract to settle against; cross-pool sharing |
| Token movement | Only net swap delta | add + remove modifyLiquidity cancel via flash accounting |
| Tick ranges | User-selected per-pool (V3/V4 UX) | Familiar UX; reuses V4 tick spacing |
| Aggregation | Sum all user positions sharing same (pool, tickLower, tickUpper) | 1 `modifyLiquidity` call per range, not per user |
| Fee tracking & Amp | Per-user perfect PnL tracking | True Liquidity Amplification achieved via exact `BalanceDelta` distribution |

### True Liquidity Amplification (No Socialization)
The `SharedLiquidityPool` utilizes an exact mathematical model to achieve true liquidity amplification where 1 ETH of real capital can back an infinite amount of virtual positions across multiple pools simultaneously, **without any socialization of losses or fees.**

**The Mechanic:**
1. **Unrealized Positions:** `addPosition` logs the user's intent to inject virtual liquidity alongside their initial backing capital. It does not deduct `freeBalance`.
2. **Dynamic Range Scaling:** During `preSwap`, the pool iterates over all active users in an about-to-be-swapped range. It checks their real-time `freeBalance` against their initial liability and calculates an individualized capacity `scale`.
3. **Perfect PnL Tracking:** During `postSwap`, the hook provides the exact `BalanceDelta` generated by the V4 pool. The engine computes `netPnL = mintDelta + burnDelta` and distributes this exact PnL to overlapping users based on their localized scale, perfectly tracking IL and exact Swap Fees without socialized accounting.

### Hook Address Constraint

V4 hooks must have specific permission bits set in the lowest 14 bits of the contract address.
- `Aqua0Hook` requires `BEFORE_SWAP_FLAG | AFTER_SWAP_FLAG = 0xC0`
- Deployed via CREATE2 + salt mining in `script/DeployV4Hookathon.s.sol` using `HookMiner.sol`

## 9. Debugging Protocol

> All bugs are tracked in **`hookathon-context/debugging-issues.md`**.
> Agents must update that file when finding or fixing any issue.

### When investigating a new revert:
1. `cast run <txhash> --rpc-url http://localhost:8545` — full execution trace
2. `cast 4byte <selector>` — decode custom error names
3. Check transient state layout (CurrencyDelta uses `keccak256(target ++ currency)`)
4. Fix, restart `./scripts/start-local.sh`, verify with `cast send`
